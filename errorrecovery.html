9. Help Users Recognize, and Diagnose from Errors

"Error messages should be expressed in plain language (no codes), precisely indicate the problem, and constructively suggest a solution".

This error message does an effective job clearly illustrating that an error has occurred. As far as recovering from an error, we believe that this method is satisfactory.

Within this form, the recovery from the error is presented in a way that forces the user to resort to browser-based functions to go back. Is this the best convention for usability in website design?

This error message displayed when we failed to enter login information to view the list of courses. We are forced to encounter this error message because we do not have this information and have to press cancel. A simple "requires login" on the original page would be very helpful.

This error message occurred due to a bad pathway. The "Contact Us" link that we clicked from the UL Lafayette College of Arts program page is a dead link, although we assumed that it would help us get in touch with someone at the College of Arts.

The error page offers links back to the page we came from as well as the homepage, index, and search tool. It also requests that we let someone know about the problem that occurred, but we have no idea who the author of "that page" is, or how to find it--after all, we were looking for some contact information in the first place.

We noticed that the e-mail for the webmaster is different from the one in the footer, which makes us wonder if this page itself is outdated.
